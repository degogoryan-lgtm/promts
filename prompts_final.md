# Система промптов для генерации артефактов аналитики
## Инвест. платформа (ЛК Брокера)

---

## Как пользоваться этим документом

Промпты используются последовательно — каждый следующий
опирается на результат предыдущего.

Порядок работы:
1. Отправь промпт в LLM
2. Дождись Шага 1 — проверь результат глазами
3. Напиши "Подтверждаю" только если всё верно
4. Получи Шаг 2 — финальный артефакт

Не переходи к следующему промпту пока не закончен предыдущий.

## Промпт 1 — BRD

```
Ты помогаешь мне написать BRD для брокерской
платформы (Личный кабинет Брокера).

СТРУКТУРА ДОКУМЕНТА (соблюдай строго):

1. Введение
   1.1 Цель документа
   1.2 Предпосылки и контекст
       Таблица бизнес-болей:
       № | Боль пользователя (Как сейчас) | Наша возможность (Как будет)
       Минимум 5 строк из реального контекста брокерского ЛК.
   1.3 Цели и задачи проекта
   1.4 Критерии успеха MVP
       Таблица: Критерий | Целевое значение

2. Заинтересованные стороны
   Таблица: Сторона | Роль | Интересы | Участие в проекте
   Роли: Инвестор, Трейдер, Администратор

3. Текущее положение дел (AS-IS)
   3.1 Процесс подачи торговой заявки (пошагово)
   3.2 Проблемы текущего процесса

4. Целевое состояние (TO-BE)
   4.1 Процесс подачи заявки
       Таблица: Шаг | Действие пользователя |
       Действие системы | Результат
       Показать:
       — проверку лимита (максимум 10 000 000 рублей)
       — параллельные процессы (проверка риска + расчёт комиссии)
       — переходы статусов: PENDING → PROCESSING
         → COMPLETED / FAILED
   4.2 Процесс просмотра портфеля
   4.3 Процесс вывода средств

5. Область проекта (Scope)
   5.1 Входит в MVP
   5.2 Не входит в MVP
       Таблица: Раздел | Причина исключения

6. Бизнес-требования
   Формат: BR-N | Название | Описание | Приоритет
   6.1 Требования к торговым операциям
   6.2 Требования к портфелю
   6.3 Требования к балансу и выводу средств
   6.4 Требования к уведомлениям
   6.5 Требования к безопасности
   6.6 Требования к интеграциям

7. Ограничения и допущения
   7.1 Регуляторные (39-ФЗ, требования ЦБ РФ, 54-ФЗ)
   7.2 Технические
   7.3 Допущения

8. Критерии успеха MVP

9. Приложения
   Глоссарий терминов (минимум 10 терминов)

КОНТЕКСТ ДОМЕНА:
— Максимальная сумма сделки: 10 000 000 рублей
— Валюты: RUB, USD, EUR, CNY
— Статусы заявок: PENDING, PROCESSING,
  COMPLETED, FAILED, CANCELLED
— Типы инструментов: STOCK, BOND, ETF, FUTURES
— Типы комиссий: FIXED, PERCENTAGE, MIXED

ФОРМАТ ВЫВОДА: Markdown

Работаем в два шага.
Не переходи к шагу 2 без моего подтверждения.

ШАГ 1 — Структура документа текстом

Опиши кратко каждый раздел — что в него войдёт,
какие бизнес-правила будут зафиксированы.

После вывода напиши:
"Шаг 1 готов. Проверьте структуру.
Если всё верно — напишите 'Подтверждаю'
и я перейду к написанию полного BRD."

ШАГ 2 — Полный документ BRD

Только после моего подтверждения.
Напиши полный BRD по структуре выше.

После вывода напиши:
"BRD готов. Жду подтверждения перед
```

## Промпт 2 — Sequence-диаграмма: Подача торговой заявки

```
Ты помогаешь мне написать Sequence-диаграмму
для брокерской платформы (Личный кабинет Брокера).

Диаграмма генерируется в синтаксисе
сайта sequencediagram.org.

ТИПЫ ЭЛЕМЕНТОВ:
actor Название        — пользователь (человечек)
boundary Название     — экран или интерфейс
control Название      — сервис или контроллер
entity Название       — объект данных или БД

ТИПЫ СТРЕЛОК:
A->B: текст          — синхронный вызов
A->>B: текст         — асинхронный вызов
B-->A: текст         — ответное сообщение

БЛОКИ:
alt [условие]        — альтернативные ветки
else [условие]       — следующая ветка
end
ref over A,B: текст  — ссылка на другой сценарий

СЦЕНАРИЙ: Подача торговой заявки

Участники:
actor        Инвестор
boundary     "Экран заявок"
control      "Сервис заявок"
control      "Сервис риска"
control      "Сервис комиссий"
control      "Биржевой шлюз"
control      "Сервис уведомлений"
entity       "База данных"

Что обязательно показать:
1. Инвестор открывает экран и заполняет заявку
2. Проверка лимита суммы (максимум 10 000 000 рублей)
   alt [сумма превышает лимит]
     — отказ с сообщением об ошибке
   else [сумма в пределах лимита]
     — продолжение
   end
3. Параллельная проверка риска (Сервис риска)
   и расчёт комиссии (Сервис комиссий)
4. Отправка заявки на биржу (Биржевой шлюз)
   alt [биржа подтвердила]
     — статус меняется на COMPLETED
   else [биржа отклонила]
     — статус меняется на FAILED
   end
5. Запись результата в базу данных
6. Асинхронное уведомление инвестора
   через Сервис уведомлений

Работаем в два шага.
Не переходи к шагу 2 без моего подтверждения.

ШАГ 1 — Описание сценария текстом

Опиши пошагово что происходит в сценарии —
кто к кому обращается, какие данные передаются,
какие ветки возможны.

После вывода напиши:
"Шаг 1 готов. Проверьте логику сценария.
Если всё верно — напишите 'Подтверждаю'
и я перейду к генерации кода."

ШАГ 2 — Код для sequencediagram.org

Только после моего подтверждения.

Правила:
— названия участников без косых черт и спецсимволов
— если в названии есть пробел — бери в кавычки
— все ответные сообщения через -->
— все асинхронные вызовы через ->>
— alt/else/end для всех ветвлений
— комментарии через note over участник: текст

После вывода напиши:
"Код готов. Скопируйте его на сайт
sequencediagram.org и нажмите Refresh."
```

## Промпт 5 — ER-диаграмма

```

ОБЯЗАТЕЛЬНЫЕ ТАБЛИЦЫ:

— accounts (счета)
— instruments (торговые инструменты)

— portfolios (портфель)

Название таблицы
PK  название_поля   тип

    название_поля   тип

"Шаг 1 готов. Проверьте сущности и связи.

Только после моего подтверждения.

Формат каждой таблицы:
— прямоугольник с заголовком (название таблицы)
— внутри список полей: PK/FK + название + тип

Формат связей:
— линия между таблицами
— на конце "один" — цифра 1
— на конце "много" — цифра N

— таблицы не должны перекрываться
— связанные таблицы располагай рядом
```

## Промпт 6 — OpenAPI (YAML для Swagger)

```
Ты помогаешь мне написать OpenAPI спецификацию
для брокерской платформы (Личный кабинет Брокера).

Используй версию OpenAPI 3.0.3.
Формат вывода — YAML.

СТИЛЬ (строго соблюдай):

1. Все переиспользуемые элементы выноси в components/:
   — schemas — схемы запросов и ответов
   — parameters — заголовки и query-параметры
   — responses — типовые ошибки (401, 403, 404, 429)

2. Примеры пиши прямо в YAML через example:

3. Описания на русском языке.

4. Бизнес-коды ошибок в поле code:
   LIMIT_EXCEEDED, INSUFFICIENT_FUNDS,
   COMPLIANCE_REJECTED, DUPLICATE_ORDER

5. Заголовки:
   — Authorization: Bearer JWT — на всех эндпоинтах
   — Idempotency-Key — на всех POST
   — If-Match — на PATCH и операциях изменения статуса
   — If-None-Match — на GET с кэшированием
   — ETag — в ответах GET
   — Cache-Control — в ответах GET
   — Retry-After — в ответах 429 и 503

6. Правила для типов данных:
   — идентификаторы: type: string, format: uuid
   — деньги: type: number, format: decimal
   — даты: type: string, format: date-time
   — статусы: type: string, enum: [...]

7. Никогда не ставь $ref рядом с другими полями.
   Только $ref или только описание — не вместе.

БАЗОВЫЕ ПАРАМЕТРЫ:
openapi: "3.0.3"
info:
  title: Broker Platform API
  version: "1.0.0"
servers:
  - url: https://api.broker.ru/v1
    description: Production
security:
  - BearerAuth: []

КОНТЕКСТ ДОМЕНА:
— Максимальная сумма сделки: 10 000 000 рублей
— Валюты: RUB, USD, EUR, CNY
— Статусы заявок: PENDING, PROCESSING,
  COMPLETED, FAILED, CANCELLED
— Типы инструментов: STOCK, BOND, ETF, FUTURES
— Типы заявок: MARKET, LIMIT, STOP
— Направления: BUY, SELL
— Типы комиссий: FIXED, PERCENTAGE, MIXED

ОБЯЗАТЕЛЬНЫЕ ЭНДПОИНТЫ:

Торговые заявки:
  POST   /v1/orders
  GET    /v1/orders
  GET    /v1/orders/{id}
  DELETE /v1/orders/{id}

Портфель:
  GET    /v1/portfolio
  GET    /v1/portfolio/performance

Баланс и вывод:
  GET    /v1/accounts/{id}/balance
  POST   /v1/withdrawals
  GET    /v1/withdrawals/{id}

Транзакции:
  GET    /v1/transactions
  GET    /v1/transactions/{id}

ДЛЯ КАЖДОГО ЭНДПОИНТА УКАЗАТЬ:
— summary — одна строка что делает метод
— description — подробнее с бизнес-правилами
— parameters — все заголовки и query-параметры
— requestBody — для POST и PATCH
— responses — все коды ответов включая ошибки
— example — пример запроса и ответа в YAML

КОДЫ ОТВЕТОВ:
200 / 201 — успех
400 — невалидный запрос
401 — не авторизован
403 — нет прав
404 — не найдено
422 — бизнес-ошибка
429 — превышен rate limit
500 — внутренняя ошибка

СПЕЦИАЛЬНЫЕ ТРЕБОВАНИЯ:

POST /v1/orders:
— Idempotency-Key обязателен
— Валидация: amount <= 10000000
— 422 с кодом LIMIT_EXCEEDED при превышении
— 422 с кодом INSUFFICIENT_FUNDS

GET /v1/portfolio:
— ETag и Cache-Control в ответе
— If-None-Match в запросе
— 304 Not Modified если кэш актуален

POST /v1/withdrawals:
— 422 с кодом COMPLIANCE_REJECTED
— 422 с кодом INSUFFICIENT_FUNDS
```
