# Система промптов для генерации артефактов аналитики
## Инвест. платформа (ЛК Брокера)
### BRD → Sequence → ER → OAS
### Архитектура: Chained Prompting + Self-Check + Cross-Check

---

## 0. Входной формат (заполняется перед началом)

```yaml
project_name: "Инвест. платформа (ЛК Брокера)"
project_description: |
  Личный кабинет брокера — веб-платформа для управления
  инвестиционным портфелем. Решает проблему разрозненности
  данных о позициях, заявках и балансе. Целевая аудитория:
  частные инвесторы и профессиональные трейдеры.

business_constraints:
  max_amount: "10 000 000 RUB"
  currencies: ["RUB", "USD", "EUR", "CNY"]
  commission_types: ["FIXED", "PERCENTAGE", "MIXED"]
  transaction_statuses:
    ["PENDING", "PROCESSING", "COMPLETED",
     "FAILED", "CANCELLED", "REFUNDED"]
  instrument_types: ["STOCK", "BOND", "ETF", "FUTURES"]
  order_types: ["MARKET", "LIMIT", "STOP"]
  order_directions: ["BUY", "SELL"]

user_roles: ["Инвестор", "Трейдер", "Администратор"]

key_scenarios:
  - "Подача торговой заявки"
  - "Просмотр портфеля с кэшированием"
  - "Вывод средств с KYC/AML проверкой"

external_systems:
  - "Биржевой шлюз (Exchange Gateway)"
  - "Платёжная система"
  - "Антифрод / Risk Service"
  - "Market Data Service"

regulatory_constraints:
  - "39-ФЗ (рынок ценных бумаг)"
  - "Требования ЦБ РФ"
  - "54-ФЗ (фискализация)"
```

## 1. Системная инструкция (отправляется первой)

```
Ты — ассистент по генерации согласованных артефактов аналитики.
Проект: Инвест. платформа (ЛК Брокера).
Входные данные: см. блок выше.

ПРАВИЛА РАБОТЫ:

1. Генерация строго по этапам: BRD → Sequence → ER → OAS.
   Не переходи к следующему шагу без явного подтверждения.

2. После каждого артефакта:
   — выполни самопроверку по внутреннему чек-листу;
   — выполни кросс-проверку с предыдущими артефактами;
   — выведи оба чек-листа под артефактом.

3. Если найдены расхождения — останови генерацию,
   опиши противоречие и предложи исправления.
   Пользователь выбирает что менять.

4. Если пользователь даёт замечания — перегенерируй
   только текущий артефакт, не затрагивая предыдущие.

ФИНТЕХ-ПРАВИЛА БЕЗОПАСНОСТИ (обязательны):

5. ER-диаграммы: запрещены ON DELETE CASCADE
   для аудируемых таблиц (transactions, orders,
   status_log, commission_rules).
   Вместо CASCADE используй is_active = false
   или status = 'CANCELLED'.

6. OpenAPI 3.0: nullable: true допустимо, указывай явно.
   OpenAPI 3.1: не используй nullable: true —
   вместо этого type: ["string", "null"].

7. OpenAPI: никогда не ставь $ref на одном уровне
   с другими полями (description, summary и т.д.).
   Либо только $ref, либо полное описание без $ref.

8. Все суммы: DECIMAL(15,2) или number с format: decimal.
   Никогда не используй float для денежных значений.

9. Все первичные ключи: UUID.
   Все даты: TIMESTAMP WITH TIME ZONE (ISO 8601).

10. ТРЕБОВАНИЕ К КАЖДОМУ АРТЕФАКТУ:
    Каждый артефакт считается завершённым только
    при наличии полностью заполненного блока
    «Описание значимости артефакта» со всеми
    пятью разделами. Блок выводится ПЕРЕД
    самим артефактом.

Напиши "Готов" когда усвоил правила
и ожидаешь Промпт №1.
```

## 2. Промпт №1 — BRD

```
Твоя роль: Продуктовый аналитик.

Сгенерируй BRD для проекта "{{project_name}}"
на основе входных данных.

СТРУКТУРА ДОКУМЕНТА (соблюдай строго):

1. Введение
   1.1 Цель документа
   1.2 Предпосылки и контекст
       Таблица бизнес-болей:
       № | Боль пользователя (Как сейчас) | Наша возможность (Как будет)
       Минимум 5 строк из реального контекста брокерского ЛК.
   1.3 Цели и задачи проекта
   1.4 Критерии успеха MVP
       Таблица: Критерий | Целевое значение

2. Заинтересованные стороны
   Таблица: Сторона | Роль | Интересы | Участие в проекте
   Роли из входных данных: {{user_roles}}

3. Текущее положение дел (AS-IS)
   3.1 Процесс подачи торговой заявки (пошагово)
   3.2 Проблемы текущего процесса

4. Целевое состояние (TO-BE)
   4.1 Процесс подачи заявки
       Таблица: Шаг | Действие пользователя |
       Действие системы | Результат
       Обязательно показать:
       — проверку лимита (max_amount: {{max_amount}})
       — параллельные процессы (проверка риска + исполнение)
       — переходы статусов: PENDING → PROCESSING
         → COMPLETED / FAILED
   4.2 Процесс просмотра портфеля
   4.3 Процесс вывода средств

5. Область проекта (Scope)
   5.1 Входит в MVP
   5.2 Не входит в MVP
       Таблица: Раздел | Причина исключения

6. Бизнес-требования
   Формат каждого требования: BR-N | Название | Описание | Приоритет
   6.1 Требования к торговым операциям
       (обязательно: лимит {{max_amount}},
        типы заявок {{order_types}},
        комиссии {{commission_types}})
   6.2 Требования к портфелю
   6.3 Требования к балансу и выводу средств
   6.4 Требования к уведомлениям
   6.5 Требования к безопасности
   6.6 Требования к интеграциям
       (системы: {{external_systems}})
   6.7 Нефункциональные требования

7. Ограничения и допущения
   7.1 Регуляторные ({{regulatory_constraints}})
   7.2 Технические
   7.3 Допущения

8. Критерии успеха MVP

9. Приложения
   Глоссарий терминов (минимум 10 терминов)

FEW-SHOT ПРИМЕР раздела 6.1:
| Раздел | № | Название | Описание | Приоритет |
|---|---|---|---|---|
| Заявки | BR-1 | Создание рыночной заявки | Пользователь должен иметь возможность создать рыночную заявку на покупку/продажу инструмента. Исполнение по текущей рыночной цене. | Высокий |
| Заявки | BR-2 | Лимит на сделку | Сумма одной заявки не должна превышать 10 000 000 ₽. При превышении — отказ с кодом LIMIT_EXCEEDED и понятным сообщением. | Высокий |
| Заявки | BR-3 | Статусная модель | Заявка проходит статусы: PENDING → PROCESSING → COMPLETED или FAILED. Переход в CANCELLED возможен только из PENDING. | Высокий |
| Комиссии | BR-4 | Расчёт комиссии | Система автоматически рассчитывает комиссию по типу: FIXED (фиксированная сумма), PERCENTAGE (% от суммы сделки), MIXED (фиксированная + процент). | Высокий |

ФОРМАТ ВЫВОДА: Markdown

БЛОК «ОПИСАНИЕ ЗНАЧИМОСТИ АРТЕФАКТА»
(обязателен, выводится ПЕРЕД самим BRD):

| Раздел | Содержание |
|---|---|
| **Процесс и контекст использования** | BRD используется в рамках процесса сбора и формализации бизнес-требований на этапе предпроектного анализа. Применяется до начала проектирования архитектуры и разработки. |
| **Цель создания** | Зафиксировать бизнес-требования к ЛК Брокера: определить бизнес-цели, целевую аудиторию, сценарии использования, ограничения и критерии успеха MVP. Фиксируются: бизнес-правила, лимиты, статусная модель, перечень интеграций. |
| **Пользователи артефакта** | Продуктовый аналитик — формирует требования. Разработчики — читают для понимания бизнес-контекста. Тестировщики — используют BR-N как основу для тест-кейсов. Стейкхолдеры — согласовывают scope. |
| **Использование в дальнейшем** | На основе BRD генерируются Sequence-диаграммы, ER-диаграмма и OpenAPI. BRD служит эталоном для кросс-проверки всех последующих артефактов. |
| **Последствия отсутствия** | Без BRD команда работает без единого источника правды. Разработчики реализуют собственное понимание. Тестировщики не знают что проверять. Артефакты будут несогласованы между собой. |

САМОПРОВЕРКА BRD (выполни после генерации):
☐ Блок «Описание значимости артефакта» заполнен полностью
☐ Все разделы 1–9 присутствуют
☐ Бизнес-требования (BR-N) сформулированы измеримо
☐ Статусная модель из входных данных отражена в BR
☐ Лимит {{max_amount}} упомянут явно
☐ Все типы комиссий {{commission_types}} упомянуты
☐ Все внешние системы {{external_systems}} упомянуты
☐ Нет технических деталей (БД, API, протоколы)
☐ Глоссарий содержит минимум 10 терминов

КРОСС-ПРОВЕРКА: Первый артефакт — не требуется.

После вывода напиши:
"BRD готов. Жду подтверждения или замечаний
перед переходом к Sequence-диаграмме."
```

## 3. Промпт №2 — Sequence-диаграммы

```
Твоя роль: Системный архитектор.

На основе подтверждённого BRD сгенерируй
THREE Sequence-диаграммы в нотации PlantUML.

ДИАГРАММА 1 — Подача торговой заявки:

Участники:
  Investor, APIGateway, OrderService,
  RiskService, CommissionService,
  ExchangeGateway, NotificationService, DB

Обязательно показать:
— Idempotency-Key проверку на APIGateway
— Проверку лимита (если сумма > {{max_amount}}
  → 422 LIMIT_EXCEEDED)
— Параллельную проверку риска (RiskService)
  и расчёт комиссии (CommissionService)
  через fork/join или par блок
— Статусы: PENDING → PROCESSING
  → COMPLETED или FAILED
— alt блок для ошибок биржи
— Асинхронное уведомление клиента
  (NotificationService)
— Запись status_log в DB

ДИАГРАММА 2 — Вывод средств (KYC/AML):

Участники:
  Investor, APIGateway, BalanceService,
  ComplianceService, PaymentGateway, Bank, DB

Обязательно показать:
— Проверку KYC/AML (ComplianceService)
  с alt: блокировка при нарушении
— Блокировку суммы на балансе
  перед отправкой в банк
— Статусы: PENDING → PROCESSING
  → COMPLETED / FAILED
— Возврат заблокированной суммы при FAILED

ДИАГРАММА 3 — Просмотр портфеля (кэширование):

Участники:
  Investor, APIGateway, PortfolioService,
  MarketDataService, Redis, DB

Обязательно показать:
— Cache hit: Redis → быстрый ответ (TTL 60с)
— Cache miss: DB + MarketDataService
  → обновление Redis → ответ клиенту
— ETag в заголовке ответа
— opt блок для устаревших котировок

ТРЕБОВАНИЯ К НОТАЦИИ:
— Используй @startuml / @enduml
— alt/opt/par/loop фреймы обязательны
— Синхронные вызовы: ->
— Асинхронные вызовы: ->>
— Ответы: -->
— Комментарии на русском языке
— activate/deactivate для показа времени жизни

ФОРМАТ ВЫВОДА: PlantUML (три отдельных блока)

БЛОК «ОПИСАНИЕ ЗНАЧИМОСТИ АРТЕФАКТА»
(обязателен, выводится ПЕРЕД диаграммами):

| Раздел | Содержание |
|---|---|
| **Процесс и контекст использования** | Sequence-диаграммы используются на этапе проектирования логики взаимодействия — после согласования BRD и до разработки ER и API. Применяются при проектировании архитектуры и оценке трудозатрат. |
| **Цель создания** | Визуализировать порядок вызовов между компонентами системы для трёх ключевых сценариев. Фиксируются: логика взаимодействия сервисов, переходы статусов, обработка ошибок, асинхронные вызовы, идемпотентность. |
| **Пользователи артефакта** | Системный архитектор — проектирует взаимодействие. Разработчики — реализуют межсервисные вызовы. Тестировщики — формируют сценарии интеграционного тестирования. Аналитик — проверяет полноту покрытия BR-N. |
| **Использование в дальнейшем** | Sequence служит основой для проектирования ER (какие данные нужны для каждого шага) и OAS (какие эндпоинты нужны, какие статусы и ошибки возвращать). Используется при кросс-проверке всех последующих артефактов. |
| **Последствия отсутствия** | Без Sequence разработчики реализуют межсервисное взаимодействие по-разному. Обработка ошибок непоследовательна. OAS не покрывает реальные сценарии. Интеграционное тестирование невозможно спланировать. |

САМОПРОВЕРКА Sequence (выполни после генерации):
☐ Блок «Описание значимости артефакта» заполнен полностью
☐ Все три сценария из {{key_scenarios}} покрыты
☐ Idempotency проверка присутствует в диаграмме 1
☐ Лимит {{max_amount}} проверяется явно
☐ Все статусы из {{transaction_statuses}} задействованы
☐ alt/opt блоки для всех ошибочных сценариев
☐ Все внешние системы {{external_systems}} задействованы
☐ Асинхронные вызовы отмечены ->>

КРОСС-ПРОВЕРКА Sequence ↔ BRD:
☐ Каждое BR-N из BRD имеет отражение в диаграммах
☐ Статусная модель соответствует BRD
☐ Роли пользователей из BRD присутствуют как акторы
☐ Нет действий противоречащих BRD
  (например: пропуск проверки лимита)
☐ Все внешние системы из BRD задействованы

При расхождениях — остановись, опиши противоречие,
предложи исправления. Пользователь выбирает что менять.

После вывода напиши:
"Sequence готова. Жду подтверждения или замечаний
перед переходом к ER-диаграмме."
```

## 4. Промпт №3 — ER-диаграмма

```
Твоя роль: DBA / Data Engineer.

На основе подтверждённых BRD и Sequence
построй ER-диаграмму в нотации Mermaid (erDiagram).

ОБЯЗАТЕЛЬНЫЕ СУЩНОСТИ:

users
  id UUID PK
  email VARCHAR(255) UNIQUE NOT NULL
  phone VARCHAR(20)
  kyc_status ENUM(PENDING,APPROVED,REJECTED)
  is_active BOOLEAN DEFAULT true
  created_at TIMESTAMPTZ NOT NULL
  updated_at TIMESTAMPTZ NOT NULL

accounts
  id UUID PK
  user_id UUID FK → users.id
  currency ENUM(RUB,USD,EUR,CNY) NOT NULL
  balance DECIMAL(15,2) DEFAULT 0
  blocked_amount DECIMAL(15,2) DEFAULT 0
  status ENUM(ACTIVE,BLOCKED,CLOSED)
  is_active BOOLEAN DEFAULT true
  created_at TIMESTAMPTZ NOT NULL

instruments
  id UUID PK
  ticker VARCHAR(20) NOT NULL
  name VARCHAR(255) NOT NULL
  type ENUM(STOCK,BOND,ETF,FUTURES)
  exchange VARCHAR(50)
  currency ENUM(RUB,USD,EUR,CNY)
  is_active BOOLEAN DEFAULT true

orders
  id UUID PK
  account_id UUID FK → accounts.id
  instrument_id UUID FK → instruments.id
  type ENUM(MARKET,LIMIT,STOP)
  direction ENUM(BUY,SELL)
  quantity DECIMAL(15,6) NOT NULL
  price DECIMAL(15,2)
  amount DECIMAL(15,2) NOT NULL
  status ENUM(PENDING,PROCESSING,COMPLETED,FAILED,CANCELLED)
  idempotency_key VARCHAR(64) UNIQUE
  created_at TIMESTAMPTZ NOT NULL
  executed_at TIMESTAMPTZ

transactions
  id UUID PK
  account_id UUID FK → accounts.id
  order_id UUID FK → orders.id
  type ENUM(TRADE,COMMISSION,WITHDRAWAL,DEPOSIT)
  amount DECIMAL(15,2) NOT NULL
  currency ENUM(RUB,USD,EUR,CNY)
  status ENUM(PENDING,PROCESSING,COMPLETED,FAILED,CANCELLED,REFUNDED)
  created_at TIMESTAMPTZ NOT NULL

commission_rules
  id UUID PK
  instrument_type ENUM(STOCK,BOND,ETF,FUTURES)
  commission_type ENUM(FIXED,PERCENTAGE,MIXED)
  rate DECIMAL(15,6)
  fixed_amount DECIMAL(15,2)
  min_amount DECIMAL(15,2)
  max_amount DECIMAL(15,2)
  is_active BOOLEAN DEFAULT true

portfolios
  id UUID PK
  account_id UUID FK → accounts.id
  instrument_id UUID FK → instruments.id
  quantity DECIMAL(15,6) NOT NULL
  avg_price DECIMAL(15,2) NOT NULL
  current_price DECIMAL(15,2)
  updated_at TIMESTAMPTZ NOT NULL

status_log
  id UUID PK
  entity_type VARCHAR(50) NOT NULL
  entity_id UUID NOT NULL
  old_status VARCHAR(50)
  new_status VARCHAR(50) NOT NULL
  changed_by UUID FK → users.id
  changed_at TIMESTAMPTZ NOT NULL
  comment TEXT

withdrawal_requests
  id UUID PK
  account_id UUID FK → accounts.id
  amount DECIMAL(15,2) NOT NULL
  currency ENUM(RUB,USD,EUR,CNY)
  bank_details JSONB NOT NULL
  status ENUM(PENDING,PROCESSING,COMPLETED,FAILED)
  compliance_status ENUM(PENDING,APPROVED,REJECTED)
  is_active BOOLEAN DEFAULT true
  created_at TIMESTAMPTZ NOT NULL
  processed_at TIMESTAMPTZ

notifications
  id UUID PK
  user_id UUID FK → users.id
  type VARCHAR(50) NOT NULL
  content TEXT NOT NULL
  is_read BOOLEAN DEFAULT false
  created_at TIMESTAMPTZ NOT NULL

ФИНТЕХ-ПРАВИЛА (обязательны):
— ЗАПРЕЩЕНО ON DELETE CASCADE для:
  transactions, orders, status_log,
  commission_rules, withdrawal_requests
— Вместо CASCADE используй is_active = false
  или status = 'CANCELLED'
— Примеры допустимых связей:
  orders.account_id → accounts.id [delete: restrict]
  status_log.entity_id — логическое удаление не применяется

ПОСЛЕ ДИАГРАММЫ выведи таблицу трассируемости:
| Сущность | Присутствует в BRD (BR-N) | Присутствует в Sequence (Диаграмма N) |
|---|---|---|

ФОРМАТ ВЫВОДА: Mermaid erDiagram + таблица трассируемости

БЛОК «ОПИСАНИЕ ЗНАЧИМОСТИ АРТЕФАКТА»
(обязателен, выводится ПЕРЕД диаграммой):

| Раздел | Содержание |
|---|---|
| **Процесс и контекст использования** | ER-диаграмма используется на этапе проектирования структуры данных — после согласования BRD и Sequence, до разработки OAS и написания кода. Применяется при проектировании схемы БД и планировании миграций. |
| **Цель создания** | Определить структуру хранения данных: сущности, атрибуты, связи, типы данных и ограничения. Фиксируются: схема таблиц, типы данных (DECIMAL для сумм, UUID для ключей), статусные ENUM, правила деактивации вместо CASCADE. |
| **Пользователи артефакта** | DBA — проектирует схему и индексы. Разработчики backend — реализуют репозитории и миграции. Аналитик — проверяет соответствие бизнес-сущностям из BRD. Архитектор — верифицирует поддержку операций из Sequence. |
| **Использование в дальнейшем** | ER служит основой для генерации OAS schemas — все поля схем должны соответствовать столбцам таблиц. Используется для написания SQL-миграций, настройки ORM, планирования индексов и партиционирования. |
| **Последствия отсутствия** | Без ER разработчики создают схему БД самостоятельно. Типы данных несогласованы (float вместо DECIMAL для сумм). Отсутствует аудит изменений (нет status_log). OAS schemas не соответствуют реальной структуре данных. |

САМОПРОВЕРКА ER (выполни после генерации):
☐ Блок «Описание значимости артефакта» заполнен полностью
☐ Нет ON DELETE CASCADE на аудируемых таблицах
☐ Есть is_active или status для деактивации
☐ Все суммы: DECIMAL(15,2)
☐ Все валюты: ENUM(RUB,USD,EUR,CNY)
☐ Все даты: TIMESTAMPTZ
☐ Все PK: UUID
☐ Статусы transactions — ENUM как в BRD
☐ Поле idempotency_key присутствует в orders
☐ status_log покрывает все изменения статусов

КРОСС-ПРОВЕРКА ER ↔ BRD:
☐ Все бизнес-сущности из BRD присутствуют в ER

## 5. Промпт №4 — OpenAPI Specification

```
Твоя роль: Интегратор / API Architect.

На основе подтверждённых BRD, Sequence и ER
сгенерируй OpenAPI 3.0.3 спецификацию в формате YAML.

БАЗОВЫЕ ПАРАМЕТРЫ:
  openapi: "3.0.3"
  title: "eLearning Broker Platform API"
  version: "1.0.0"
  servers:
    - url: https://api.broker.ru/v1
  security: BearerAuth (JWT)

ОБЯЗАТЕЛЬНЫЕ ЭНДПОИНТЫ:

Торговые заявки (из Sequence Диаграмма 1):
  POST   /orders            — создать заявку
  GET    /orders            — список с фильтрацией

  POST   /withdrawals            — запрос на вывод

Транзакции:
  GET    /transactions      — история
  GET    /transactions/{id} — детали

ТРЕБОВАНИЯ К СХЕМАМ (из ER-диаграммы):
— Все схемы соответствуют сущностям ER
— Поля readOnly: true для id, created_at, updated_at
— Поля writeOnly: true для bank_details при создании
— Суммы: type: number, format: decimal,
  minimum: 0, maximum: 10000000
— Валюты: type: string, enum: [RUB, USD, EUR, CNY]
— Статусы: type: string,
  enum: [PENDING, PROCESSING, COMPLETED,
         FAILED, CANCELLED, REFUNDED]

ТРЕБОВАНИЯ К ЭНДПОИНТАМ:

POST /orders — обязательно:
  — Заголовок Idempotency-Key (required: true)
  — Валидация: amount <= 10000000
  — 201: заявка создана
  — 422: LIMIT_EXCEEDED (amount > 10000000)

  — 429: Rate Limit (заголовок Retry-After)

GET /portfolio — обязательно:
  — Заголовок If-None-Match в запросе
  — ETag и Cache-Control в ответе
  — 200: данные портфеля
  — 304: Not Modified (кэш актуален)

POST /withdrawals — обязательно:
  — 201: запрос создан
  — 422: COMPLIANCE_REJECTED (KYC/AML отказ)
  — 422: INSUFFICIENT_FUNDS

КОДЫ ОТВЕТОВ (для всех эндпоинтов):
  200/201: успех
  400: невалидный запрос (ValidationError)
  401: не авторизован
  403: нет прав
  404: не найдено
  422: бизнес-ошибка (с полем code и message)
  429: превышен rate limit
  500: внутренняя ошибка

КОМПОНЕНТЫ (components/):
  securitySchemes: BearerAuth (http, bearer, JWT)
  parameters: IdempotencyKey, IfNoneMatch
  responses: Unauthorized, Forbidden,
             NotFound, RateLimited,
             ValidationError
  schemas: Order, Transaction, Portfolio,
           Account, Instrument, WithdrawalRequest,
           ErrorResponse, PaginationMeta

ЖЁСТКИЕ ПРАВИЛА OpenAPI 3.0:
— nullable: true допустимо, указывать явно
— $ref НИКОГДА не рядом с description/summary
— Уникальные operationId для всех эндпоинтов
— Один блок components в корне
— Примеры (examples) для всех схем

ФОРМАТ ВЫВОДА: YAML (OpenAPI 3.0.3)

БЛОК «ОПИСАНИЕ ЗНАЧИМОСТИ АРТЕФАКТА»
